# 第3章：メモリ管理システム (Memory Management)

プログラミング言語におけるメモリ管理は、安全性とパフォーマンスを左右する重要な要素です。Flunoは、現代的なガベージコレクション (GC) を採用しつつ、開発者がメモリの挙動を理解しやすいような設計になっています。Rustのような複雑な生存期間（Lifetime）の指定は必須ではありませんが、効率的なリソース管理のために内部の仕組みを知っておくことは有益です。

## 3.1 ガベージコレクション (Garbage Collection)

Flunoは自動メモリ管理を採用しています。これは、プログラマが手動で `malloc` や `free` を行う必要がないことを意味します。

### トレーシングGC

Flunoのランタイムにはトレーシングガベージコレクタが組み込まれています。このGCは定期的に実行され、プログラムから到達不可能になった（どの変数からも参照されていない）オブジェクトを特定し、そのメモリを解放します。

これにより、以下の一般的なバグを防ぐことができます：
*   **Use After Free**: 解放済みのメモリにアクセスすること。
*   **Double Free**: 同じメモリを二度解放すること。
*   **Memory Leak**: 解放し忘れたメモリが蓄積すること（完全には防げませんが、循環参照などはGCが解決します）。

## 3.2 値のコピーとクローン (Copy and Clone)

Flunoの型システムでは、データの移動と複製に関して明確な区別があります。

### スタックのみのデータ：コピー (Copy)

整数型や浮動小数点型、真理値型などのサイズが固定で小さい型は、スタック上に保存されます。これらの値を変数に代入したり、関数に渡したりするときは、値そのものがビット単位でコピーされます。

```rust
let x = 5;
let y = x; // xの値がyにコピーされる

println!("x = {}, y = {}", x, y); // 両方有効
```

### ヒープデータ：クローン (Clone)

配列や構造体、あるいは大きなデータ構造はヒープメモリを使用する場合があります。これらのデータを複製コストが高いものとして扱う場合、明示的な `.clone()` 呼び出しが必要になることがあります（※ 現在の実装ではGC管理下にあるため、多くの場合は参照のコピーとなりますが、意味的に「別の独立した値」を作成する場合はクローンを用います）。

## 3.3 参照と借用 (References and Borrowing)

FlunoにはRustのような厳格な借用チェッカーはありませんが、効率のために大きなデータを関数に渡す際は「参照」として渡すことが推奨されます。

(※ 注：現在のFluno仕様では、基本的にすべてのオブジェクトは参照として扱われるか、GC管理下のポインタとして実装されています。ユーザーはポインタ演算を意識する必要はありません)

```rust
fn calculate_length(s: String) -> Int {
    s.len()
}
```

将来的には、より細かいメモリ制御のために、明示的な所有権システムがオプションとして導入される可能性がありますが、Ver 2.0時点では**「GCによる安全で楽なメモリ管理」**がFlunoの基本方針です。
